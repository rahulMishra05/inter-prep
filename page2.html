Fundamentals

Vertical vs Horizontal Scaling

Vertical: bigger box (more CPU/RAM). Easy but bounded and a single point of failure.

Horizontal: more boxes. Requires statelessness, load balancing, distributed cache and shared storage.

Rule of thumb: Scale vertically up to a comfortable ceiling, then go horizontal. Most production systems are hybrids.

Caching Strategies

  • Cache-Aside (Lazy): app reads cache, falls back to DB, writes cache. Most common.
  • Read-Through: cache service handles DB miss internally.
  • Write-Through: writes go to cache + DB synchronously. Slower writes, fresh reads.
  • Write-Behind: writes to cache, async to DB. Fast but risk of data loss.
  • Refresh-Ahead: proactively refresh hot keys before expiry.
Cache invalidation is one of the two hardest problems in CS. Prefer short TTLs + explicit invalidation on writes; consider versioned keys for hot paths.

DB Scaling Patterns

Read Replicas: async-replicate primary, route SELECTs to replicas. Caveat: replica lag.

Sharding: horizontal partitioning by key (user_id, region). Hard to re-shard later.

Vertical Partitioning: split tables across DBs by domain.

CQRS: separate read and write models. Pairs with event sourcing.

✓ Sharding Pros
  • Linear scale
  • Smaller indexes
  • Fault isolation
✗ Sharding Cons
  • Cross-shard joins are painful
  • Re-sharding is expensive
  • Hot keys cause skew

CAP Theorem

In a partitioned distributed system you must pick: Consistency or Availability. Network partitions are not optional, they happen.

  • CP: SQL Server with sync replication, MongoDB with majority writes, etcd
  • AP: Cassandra, DynamoDB (default), CouchDB
  • PACELC: extends CAP — Even when no Partition, choose between Latency and Consistency
Real talk: Most apps need strong consistency for money/auth, eventual consistency for everything else (likes, feeds, search).

Microservices vs Monolith

✓ Microservices win when
  • Multiple teams owning bounded contexts
  • Different scale needs per service
  • Independent deployment frequency
  • Polyglot tech stacks needed
✗ Microservices hurt when
  • Small team, single domain
  • No CI/CD maturity
  • Distributed transactions everywhere
  • No observability stack
Most "microservice" failures are actually distributed monoliths — services tightly coupled by sync calls and shared DBs.

Three Pillars of Observability

  • Logs: Serilog → Seq / Elastic / App Insights. Structured, correlation-id-tagged.
  • Metrics: Prometheus + Grafana / App Insights. RED method (Rate, Errors, Duration).
  • Traces: OpenTelemetry → Jaeger / Zipkin / App Insights. End-to-end request flow.
Always emit a correlation ID from the edge and propagate it via headers across services.
